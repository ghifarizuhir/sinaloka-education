Integration Architecture Approaches

How to structure the API layer and state management for sinaloka-platform

A

Layered Services Architecture ★ Recommended

Structure: Axios instance → Service modules (per domain) → React Query hooks → Components

src/
  lib/api.ts — Axios instance, interceptors, token refresh
  services/auth.service.ts — login, logout, refresh
  services/students.service.ts — CRUD calls
  services/...service.ts — one per domain
  hooks/useStudents.ts — React Query wrappers
  hooks/use...ts — one per domain
  contexts/AuthContext.tsx — token state, user info

Pros

  • Clear separation of concerns
  • Services are testable without React
  • Hooks encapsulate query keys & cache logic
  • Easy to add new domains (copy pattern)

Cons

  • More files to create upfront
  • Slightly more boilerplate per domain
B

Hooks-Only Architecture

Structure: Axios instance → React Query hooks (contain API calls directly) → Components

src/
  lib/api.ts — Axios instance, interceptors, token refresh
  hooks/useStudents.ts — API calls + React Query combined
  hooks/use...ts — one per domain
  contexts/AuthContext.tsx — token state, user info

Pros

  • Fewer files — less boilerplate
  • API call + caching logic in one place
  • Faster to scaffold

Cons

  • Harder to test API calls without React
  • Hooks can get large with CRUD + filters
  • Mixing concerns (HTTP + caching)
C

Generic CRUD Factory

Structure: A factory function generates services + hooks for any entity from a config object

src/
  lib/api.ts — Axios instance
  lib/crud-factory.ts — createCrudHooks('students', '/api/admin/students')
  hooks/useStudents.ts — = createCrudHooks(...) + custom overrides
  contexts/AuthContext.tsx — token state

Pros

  • Minimal duplication — DRY across domains
  • New entity = one line of config
  • Consistent patterns enforced

Cons

  • Abstraction can be hard to customize
  • Harder to debug — magic behind the factory
  • Each entity has unique quirks (CSV, bulk ops)
